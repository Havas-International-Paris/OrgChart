# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Internal org chart tool for Havas International: an editable spreadsheet (left) and a visual org chart (right) on one screen, backed directly by Supabase (Postgres + Auth + Realtime + Storage) with no custom backend server. Supports multi-reporting (an employee can have several managers, one flagged primary), per-employee client/mission assignments with %ETP, and multiple independent org charts within the same project.

## Commands

```bash
npm run dev       # start Vite dev server (http://localhost:5173)
npm run build     # tsc -b (typecheck) && vite build — this is the primary correctness check, run it after any change
npm run lint      # oxlint
npm run preview   # preview a production build
```

There is no test suite in this repo. `npm run build` (which runs `tsc -b` first) is the main way to catch mistakes before manual verification.

The dev server requires `.env.local` (copy from `.env.example`) with `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` from a Supabase project that has had the migrations in `supabase/migrations/` applied (see README.md for the full setup sequence — order matters, `0004` in particular is easy to forget and silently breaks realtime). Without `.env.local`, the app renders a "Configuration Supabase requise" screen instead of crashing (see `isSupabaseConfigured` in `src/lib/supabaseClient.ts`).

## Architecture

**Data flow: services → hooks → components.** `src/services/*` are thin wrappers around `supabase-js` calls (one file per table). `src/hooks/use*` each own one table's full dataset: fetch on mount, subscribe to Postgres changes, and re-`fetch()` (not incremental patching) on any change. Because the whole org is a few hundred rows at most, every hook loads its entire table and consumers filter/derive in memory (e.g. `useReportingGraph().managersOf(employeeId)`, `useAssignments().totalEtpOf(employeeId)`) rather than querying per-employee. Don't add pagination or per-entity fetching without a real scale reason.

**Realtime channel names must be unique per mount.** Every hook subscribes with `` `table-changes-${crypto.randomUUID()}` `` rather than a fixed string. A fixed channel name breaks the moment two consumers of the same hook are mounted at once (e.g. `EmployeeGrid` and `OrgChartView` both call `useEmployees()`) or React StrictMode double-mounts in dev — Supabase's realtime client throws `cannot add postgres_changes callbacks ... after subscribe()` because the second `.channel(sameName)` call returns the already-subscribed channel object. If you add a new data hook, follow this pattern.

**`employees`/`reporting_relationships`/`assignments`'s `postgres_changes` subscriptions are deliberately unfiltered** (no `filter: org_chart_id=eq.X`), relying on `refresh()`'s own query — which *is* scoped by `org_chart_id` — to keep displayed data correct. This looks wasteful (every mounted consumer refetches on any change to the table, across every org chart, not just its own) but is deliberate: a filter on a non-primary-key column like `org_chart_id` silently drops DELETE events. Postgres's default replica identity only ships primary-key columns in a DELETE's WAL entry, so Realtime can't evaluate a non-PK filter and never delivers the event to filtered subscribers — INSERT/UPDATE were never affected since the full new row is always available. Hit as a real bug (deleting an employee from the grid left a stale card in the chart, and vice versa, until a manual reload) and initially "fixed" with `alter table ... replica identity full` (`0012_realtime_delete_replica_identity.sql`) — that turned out **insufficient on its own** (verified live, the bug still reproduced after applying it), so the filter was removed entirely instead. The migration is harmless to leave applied but isn't what actually fixes this — don't reintroduce a `filter: org_chart_id=eq...` on these three subscriptions without re-verifying DELETE propagation live in the browser, not just trusting `REPLICA IDENTITY FULL` to be enough.

**Multi-reporting model.** `reporting_relationships` rows have `employee_id`, `manager_id`, `is_primary`. At most one primary manager per employee (partial unique index in `0001_init_schema.sql`). Primary edges form the tree used for the automatic dagre layout (`src/components/chart/layoutEngine.ts`); secondary edges are drawn afterward as a dashed overlay using the positions dagre already computed — they never participate in layout. `useVisibleGraph` walks the primary-edge tree from roots (employees with no primary manager) down through whichever nodes are in the `expandedNodeIds` set.

**Multiple independent org charts share the same tables.** `org_charts` (`0009_org_charts.sql`) is the scoping table; `employees`, `reporting_relationships`, and `assignments` each carry a required `org_chart_id`, and every hook/service touching those three tables takes `currentOrgChartId` (from `selectionStore`) as a parameter — `useEmployees(currentOrgChartId)`, `useReportingGraph(currentOrgChartId)`, `useAssignments(currentOrgChartId)`. `clients_missions`, `job_titles`, and `departments` stay global/unscoped catalogs shared across every chart. Uniqueness constraints (`uq_one_primary_manager`, `uq_employee_client_mission`) and the cycle-check trigger below are all re-scoped to include `org_chart_id`, so two charts can never interfere with each other even with identical employee/relationship shapes. Switching charts (the selector in `AppShell.tsx` → `setCurrentOrgChartId`; `OrgChartManagerModal.tsx` handles create/rename/duplicate/delete) resets every chart-relative field in `selectionStore` (selection, expand/focus sets, search, assignments modal). `duplicate_org_chart(source_id, new_name, new_short_label)` (a Postgres RPC) deep-copies one chart's employees/relationships/assignments into a new chart via a temp id-remap table; `clients_missions` references are copied as-is since that catalog is shared.

**Cycle prevention is deliberately doubled.** Client-side `wouldCreateCycle` (in `useReportingGraph.ts`) does a BFS from the proposed manager upward through existing edges before any write is attempted, so the UI can grey out invalid choices live. `0003_cycle_check_function.sql` adds the same check as a Postgres trigger with a recursive CTE, because the client-side check alone leaves a race window between two simultaneous users. Both must stay in sync if the reporting model changes.

**Cross-view state lives in one Zustand store**, not component state: `src/stores/selectionStore.ts` holds `selectedEmployeeId`, `searchQuery`, `expandedNodeIds`, `focusedNodeIds`, and `assignmentsEmployeeId`. `EmployeeGrid` and `OrgChartView` both read/write it so that clicking a row centers the corresponding chart node (and vice versa), and so search highlights/auto-expands ancestors in the chart while filtering the grid via AG Grid's `quickFilterText`. `AssignmentEditorModal` is rendered once at the `AppShell` level (not inside the grid or the chart) and opens based on `assignmentsEmployeeId`, so either view can trigger the same modal instance.

**A second, deliberately separate Zustand store holds per-browser UI preferences**: `src/stores/uiPreferencesStore.ts` (`gridDensity`, `splitFraction`) is wrapped in zustand's `persist` middleware, writing to `localStorage` — unlike everything in `selectionStore.ts`, which is in-memory only and explicitly wiped on every org-chart switch. These preferences never touch Supabase, so one person's density/split choice can never leak to anyone else, at the cost of not following a user across browsers/devices (an explicit, accepted tradeoff — see git history if a per-account version is ever wanted). `src/lib/gridTheme.ts`'s `getGridTheme(density)` (an AG Grid v36 `themeQuartz.withParams(...)` variant — row/header padding and font size only) and `scaleColumnWidth(px, density)` (a flat 0.75× applied to individual colDefs' `width`/`minWidth`, since the Theming API params have no effect on column pixel widths) are how the 4 AG Grid files (`EmployeeGrid`, `ClientsMissionsGrid`, `JobTitlesGrid`, `DepartmentsGrid`) each render themselves densely. `AppShell.tsx`'s grid/chart split divider is a plain `setPointerCapture`-based drag (same pattern as `PhotoEditorModal.tsx`'s pan/zoom), clamped to `[0.2, 0.75]`.

**Dagre lays out every employee, not just the currently-visible ones.** `OrgChartView.tsx`'s `layoutedNodeById` memo runs `layoutWithDagre` over the *entire* `employees`/`primaryEdges` set and depends on nothing else — not `expandedNodeIds`, not focus mode, not hover/selection/search/the dept filter. A second, cheap memo filters that fixed position map down to whichever employees are currently visible and layers on per-render styling (dimming, chain highlight, selection). This split exists because dagre is a global, non-incremental layout — re-running it on a smaller or larger node set reflows *every* node's x/y, not just the ones that appeared or disappeared, so a naive "layout only what's visible" approach makes collapsing a team or using focus mode visibly shift unrelated, still-visible cards (a real regression hit and fixed once already). Keep this split if you touch this code: anything that changes which cards render belongs in the second memo, never the first.

**The chart's initial fit is imperative, not the declarative `fitView` prop.** `<ReactFlow fitView>` only fits once, on React Flow's own mount — which happens before `employees`/`relationships` finish their async load, so it would fit an empty/stale node set and never refit once real data arrived (a chart that loads tiny or mis-cropped, inconsistently, depending on network timing). `OrgChartView.tsx` instead calls `instance.fitView()` itself inside a `useEffect` gated on both loading flags and `expandedNodeIds` being non-empty (i.e. the default-expand effect has already run), guarded by `hasAutoFitRef` so it only fires once per chart — reset when `currentOrgChartId` changes so switching charts re-fits instead of keeping the previous chart's pan/zoom.

**Hovering or pinning (clicking) a card highlights its full reporting chain.** `useReportingChain.ts` computes, for one "active" employee id, everyone to keep at full opacity (ancestors up the primary tree, any dotted/secondary managers plus *their* ancestor chain, the full descendant subtree, and anyone dotted-reporting directly to them) plus a narrower `chainIds` set used only to decide whether an *edge* counts as part of the chain. `OrgChartView.tsx` combines `hoverEmployeeId` (local state, cleared on mouse-leave) with `selectedEmployeeId` (from the store, persists until deselected) into one `activeEmployeeId` — hover takes priority while active, falling back to the pin once the mouse leaves. Clicking a card also opens `EmployeeDetailPanel.tsx` (manager/functional managers/direct reports/incoming functional reports, each row clickable to jump to that person) and pans the view to it (current zoom preserved via `getZoom()`, never forced) — but only on an actual selection change, never from toggling that same card's collapse or focus badge (`lastCenteredIdRef` in `OrgChartView.tsx` guards this).

**Two independent collapse mechanisms per card, both round edge badges** (`CollapseBadge` in `EmployeeNode.tsx`). The bottom badge toggles the person's own entry in `expandedNodeIds` (downward: hide/show their subtree — the same set `useVisibleGraph` walks). The top badge toggles their entry in `focusedNodeIds` (upward "focus mode": hide everyone *except* this person and their already-visible subtree — ancestors and unrelated branches alike disappear). `focusedNodeIds` is a Set, so multiple people can be focused at once (the visible set becomes the union of each focused person's subtree); the filter only recurses into a child via `childrenOf` if that child is *already* in the expand-based visible set, so focus mode only ever hides, never force-reveals a separately-collapsed subtree.

**"+" buttons on chart nodes** (`EmployeeNode.tsx`, top-right/bottom-right corners) each open a small popover with two actions: create a brand-new employee and link them immediately (`quickAddManager`/`quickAddSubordinate` in `OrgChartView.tsx`), or link an existing employee via `LinkExistingEmployeeModal` (candidates filtered by `wouldCreateCycle`). When linking/creating a manager relationship, whether the new edge is primary is decided automatically: primary if the employee doesn't already have one, secondary otherwise. Double-clicking a node's first name, last name, poste, or department (same file) opens an inline editor (text input for name, a `<select>` sourced from `useJobTitles()`/`useDepartments()` for poste/department, matching the grid's catalog-only constraint) — a single click keeps its existing role of selecting the node.

**Employee photos live in a public Supabase Storage bucket** (`employee-photos`, `0010_employee_photos.sql`), referenced by `employees.photo_path`, with an interactive pan/zoom crop stored as `photo_zoom`/`photo_pan_x`/`photo_pan_y` (`0011_photo_reframe.sql`, phase 5b of the design handoff). `employeePhotoService.ts` is the only place that touches `supabase.storage` — upload (with image-type/5MB checks), delete, and `employeePhotoUrl()` (a `getPublicUrl` wrapper). `usePhotoActions.ts` composes that with the DB writes (`replacePhoto` uploads then best-effort-deletes the old object; `saveFrame` persists a crop; `deletePhoto` nulls `photo_path` and resets the crop). A new upload always resets `photo_zoom`/`photo_pan_x`/`photo_pan_y` to their defaults (`employeeService.ts`'s `updateEmployeePhoto`) since the old crop was framed for a different image.

`PhotoFrame.tsx` renders the cover-fit + pan/zoom crop from those three numbers via plain CSS (explicit width/height percentages sized from the image's aspect ratio, then `transform: scale(zoom) translate(panX%, panY%)`) — no canvas re-encoding. Because every percentage is relative to a *square* container's own size rather than absolute pixels, the identical (zoom, panX, panY) triple renders correctly at any avatar size (28px grid row, 36px chart card, 220px editor preview) — `baseSizePct()` in `lib/photoFrameMath.ts` is the one place that math lives, shared by `PhotoFrame` and `PhotoEditorModal`'s own drag/zoom clamping.

`PhotoAvatar.tsx` (used by both `EmployeeNode.tsx` and `EmployeeGrid`'s photo column) is purely presentational: a single click calls a stable `onOpen(employeeId)` callback owned by the parent (`OrgChartView`/`EmployeeGrid`), which opens one `PhotoEditorModal.tsx` — browse, drag-and-drop, paste (Ctrl+V), reframe, and delete all in one interface, rather than separate hover-revealed actions. Paste is caught with a `document`-level listener while the modal is mounted, not a listener on the small avatar itself — a non-editable focused element doesn't reliably receive native paste events in every browser, and requiring the avatar to hold focus was fragile in practice. This same "keep interactive state at the parent, not in the cell" shape is required by **AG Grid's `redrawRows()`**: `EmployeeGrid.tsx` calls it (destroying and recreating every row's cell renderer components) whenever `selectedEmployeeId` changes, purely so `getRowStyle`'s selection highlight re-evaluates — a photo-editing modal or file `<input>` living inside the cell itself would get wiped by that same click before the user could finish interacting with it. `OrgChartView`/`EmployeeGrid` each hold `photoEditEmployeeId` locally and render one `PhotoEditorModal` for it.

**No AG Grid master/detail.** That's an Enterprise-only feature. Per-employee assignments (client/mission + %ETP, one-to-many) are instead edited through the standalone `AssignmentEditorModal`, opened from a summary badge/cell in both the grid and the chart — not an expandable grid row.

**`useRowStabilizer` (`src/components/grid/useRowStabilizer.ts`) keeps grid rows from jumping under the user mid-edit**, shared by all 3 left-panel grids. Two mechanisms: a per-column `comparator` freezes a row being edited at its pre-edit value so an active sort doesn't reorder it while typing; a freshly-created row is pinned to the top via AG Grid's `pinnedTopRowData` (works even with no sort active, unlike a comparator trick alone) until the user leaves it. "Left the row" is detected via a document-level capture-phase `mousedown` listener walking up from the click target — deliberately *not* `onBlur`/`relatedTarget`, which false-positives the instant an AG Grid popup editor (e.g. `agSelectCellEditor`'s dropdown) mounts/unmounts. Any modal opened from a cell while a row might be mid-edit (`ManagerEditorModal`, `AssignmentEditorModal`, `PhotoEditorModal`) must carry a `data-row-stabilizer-ignore` attribute on its root element, or every click inside it reads as "clicked outside the grid" and prematurely unpins/unfreezes the row. `popupParent={containerRef.current}` on the grid is required for the same reason, scoping AG Grid's own popups inside the tracked container.

**Left pane is tabbed** (`LeftPanel.tsx`, local `useState`, not in the Zustand store): Employés (`EmployeeGrid`), Clients / Missions (`ClientsMissionsGrid`), Allocations (`AllocationsView`, read-only, not an AG Grid at all), Postes (`JobTitlesGrid`), Business Units (`DepartmentsGrid`). A density toggle (`Confortable`/`Compact`, see `uiPreferencesStore.ts` above) sits alongside the tabs and applies to all 4 AG Grid tabs at once. `ClientsMissionsGrid` and `JobTitlesGrid` are standalone catalog editors for tables that also back other features — `clients_missions` feeds `AssignmentEditorModal`'s autocomplete, `job_titles` feeds the "Poste" column's `agSelectCellEditor` in `EmployeeGrid` (`cellEditorParams.values` sourced live from `useJobTitles()`). `job_titles` has no FK from `employees.job_title` — it's a curated suggestion list enforced only at the UI layer (select-only editor, no free text), so deleting a job title never cascades or blocks. `clients_missions` does have a real FK from `assignments` (`on delete restrict`), so `ClientsMissionsGrid`'s delete handler must catch and surface that failure.

**`src/lib/database.types.ts` is hand-authored**, not generated by the Supabase CLI, and must be kept in sync with `supabase/migrations/*.sql` manually. Once a project is linked, prefer regenerating it with `supabase gen types typescript`. It must satisfy the `GenericSchema` shape from `@supabase/supabase-js` (each table needs `Relationships: []`, the schema needs `Views`/`Functions`) or the client's generic inference silently degrades and `.insert()`/`.update()` calls fail to typecheck.

**`OrgChartView` is wrapped in an `ErrorBoundary`** (`src/components/shared/ErrorBoundary.tsx`, mounted in `AppShell.tsx`) — the only error boundary in the app. Without it, a render-time throw anywhere in the chart subtree unmounts it with no visible trace (React only logs to console), which reads as the chart silently "disappearing" rather than a diagnosable bug. Add a boundary around any other subtree where the same silent-blank-panel failure mode would be hard to notice.

**Library version notes:** AG Grid is v36 using the new Theming API (`theme={themeQuartz}` prop, no CSS imports, requires `ModuleRegistry.registerModules([AllCommunityModule])`) — don't add the old `ag-theme-*.css` imports. Tailwind is v4, configured via the `@tailwindcss/vite` plugin in `vite.config.ts`, not a `tailwind.config.js`/PostCSS setup.

## Deployment

Production: https://orgchart-dun.vercel.app, deployed by Vercel from `Havas-International-Paris/OrgChart` on GitHub — every push to `main` auto-deploys, no manual step. No separate staging environment: the same Supabase project backs both local dev and production. New SQL migrations must be applied by hand to that one project (see README.md's "Déploiement et mise à jour" section) — there's no automated migration pipeline.

**GitHub Actions + Supabase Postgres gotchas** (hit both when setting up `.github/workflows/supabase-backup.yml`, now documented inline in that file too):
- Supabase's "Direct connection" string (`db.<ref>.supabase.co`) is **IPv6-only**; GitHub's hosted runners have no IPv6 egress and fail with `Network is unreachable`. Use the **Session pooler** connection string instead (`aws-0-<region>.pooler.supabase.com`, found via the Connect modal's "Direct" tab).
- `pg_dump` refuses to dump from a server newer than itself. Supabase currently runs Postgres 17, but Ubuntu's default `postgresql-client` package is older — install `postgresql-client-17` from the official PGDG apt repo and invoke it by full path (`/usr/lib/postgresql/17/bin/pg_dump`), since the older bundled one stays first on `PATH`.
- The three repo secrets (`SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_DB_URL`) have similar names — easy to edit the wrong one via GitHub's secrets UI (it happened once already). `SUPABASE_URL` must stay a plain `https://` URL (used by the keep-alive ping); the Postgres connection string belongs only in `SUPABASE_DB_URL`.

## Known issues

- No optimistic locking: if two users edit the same field at the same moment, last write wins silently.
- No granular roles — every authenticated user can edit or delete anything.
- The weekly backup workflow (`.github/workflows/supabase-backup.yml`) only `pg_dump`s the Postgres database — it does not back up the `employee-photos` Storage bucket. Restoring from that backup would bring back every `employees.photo_path` value pointing at objects that may no longer exist.
